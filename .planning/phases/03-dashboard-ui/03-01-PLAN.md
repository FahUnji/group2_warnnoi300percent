<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input → extractProjectKey() | User types a project key or URL; value crosses from unvalidated text input into a URL path segment used in POST /api/sync/{project_key} |
| sessionStorage → React state | project key read from sessionStorage on component mount; value is used in fetch URL construction |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | connectInput → POST /api/sync/{key} | mitigate | Client-side: extractProjectKey() trims + uppercases; backend MUST validate key format (alphanumeric, max 10 chars) — backend validation is the authoritative gate. Frontend is defense-in-depth only. |
| T-03-02 | Spoofing | sessionStorage('active_project_key') | accept | sessionStorage is same-origin; no PII involved; worst case is wrong project displayed, not a security breach. |
| T-03-03 | Information Disclosure | GET /api/bugs/{project_key} response in React state | accept | Bug data is internal QA tool data; no auth credentials or PII in bug records; data visible only to authenticated user session. |
| T-03-04 | Denial of Service | Parallel fetch calls (one per project card) | mitigate | Frontend: limit parallel fetches to at most 20 via Promise.all slice in Plan 02. Backend: existing FastAPI rate limiting handles excessive requests. |
| T-03-05 | XSS via project name/key in JSX | ProjectCard renders project.name and project.key | accept | React escapes JSX text content by default; no dangerouslySetInnerHTML used; no risk. |
</threat_model>
